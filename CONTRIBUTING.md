# Contributing Policy

Thank you for your interest in THE HOOK. This repository is maintained as a defensive-security Public Edition and a controlled product-positioning surface.

## Contribution Model

External code contributions are not accepted unless explicitly approved by the repository owner in advance.

This restriction exists to protect the defensive scope of the project, avoid accidental disclosure of private enterprise methods, and keep the public repository procurement-safe, CISO-friendly, and commercially controlled.

## Contributions That May Be Considered

The maintainer may selectively consider non-sensitive contributions such as:

- typo fixes or documentation clarity improvements
- governance file improvements
- accessibility notes for public documentation
- issue reports about broken links or inaccurate public claims
- safe suggestions for public positioning or repository hygiene

Please open an issue before preparing any pull request. Unapproved pull requests may be closed without review.

## Contributions That Will Not Be Accepted

Do not submit, request, or describe:

- deception logic or operational deployment details
- telemetry pipelines, enrichment logic, or detection rules
- token generation, credential handling, secrets, API keys, or private configuration
- client environments, customer information, infrastructure diagrams, or internal network references
- private playbooks, internal prompts, restricted research, or commercial implementation details
- offensive tooling, exploit instructions, persistence techniques, evasion guidance, or unauthorized testing workflows
- changes that convert the repository from defensive-security positioning into an operational dual-use toolkit

## Public Edition Boundary

The Public Edition exists to communicate safe product direction, architecture thinking, interface quality, and governance maturity. It is not the Private Enterprise Edition and does not accept contributions that attempt to recreate private implementation details.

## Pull Request Expectations

If the maintainer explicitly invites a contribution:

1. Keep the change narrow and documentation-focused unless otherwise approved.
2. Do not modify product behavior, index.html behavior, visual design, or private implementation details unless specifically requested by the maintainer.
3. Avoid adding dependencies, scripts, data collection, telemetry, or executable behavior.
4. Keep language calm, defensive, legally careful, and enterprise appropriate.
5. Confirm that no secrets, credentials, client references, or private methods are included.

## Issues

Public issues should be limited to non-sensitive repository concerns. Do not open public issues containing vulnerability details, private architecture, implementation-sensitive details, or operational security information. Use SECURITY.md for security reporting.

## License and Ownership

All repository materials remain proprietary unless explicitly stated otherwise in LICENSE.md. Contributions, if accepted, may be incorporated into proprietary materials without creating any ownership, licensing, or commercialization rights for the contributor.

## Maintainer Discretion

The maintainer may close issues or pull requests that are out of scope, unsafe, duplicative, commercially sensitive, or inconsistent with the defensive-security purpose of this repository.
